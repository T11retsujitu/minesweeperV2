## 1. 参照PVの内容(動画はリポジトリ未収録のためテキストで記録)

10秒・1280×720・実写「手持ちスマホ」フォーマットのフェイクゲームプレイ風PV。構成:

| 秒 | 内容 |
|----|------|
| 0-1 | タイトルロゴ「マイン・ローグ」(ピクセルアート、剣の装飾)+ ダンジョン背景 |
| 2-4 | 騎士キャラがダンジョンの石畳タイル盤上に立つ。周囲のタイルが開いて数字(緑1/黄2)が出現。トラップタイル・鍵付きタイルあり |
| 5-6 | HUD(顔アイコン+HPバー185/355、サブバー50/50、コイン2352、宝石50他)。フラグ設置、宝箱、スライム敵、装備入手ポップアップ「Iron Sword +1 ★1」+Equipボタン、松明の光 |
| 7-8 | 騎士が剣でスライムを攻撃。「-5 HP」ダメージフロート、コイン飛散、対象セルの黄色ハイライト。右下に剣/トラップのアクションボタン |
| 9 | 「FLOOR CLEAR!!」祝祭演出+報酬表示 → ロゴ+「202X Coming Soon to iOS/Android」 |

### フックの正体(重要度順)

1. **「マインスイーパー×ダンジョンRPG」が1カットで伝わる構図** — 主人公の周囲にタイルが開いて数字が出る絵そのもの。既知×既知の新結合が3秒で伝わる。
2. **盤上に主人公がいる** — 感情移入の対象と空間的因果(敵が近い→危ない)が一目で読める。
   ※現在openの**課題1(被ダメージ原因が分かりにくい=プレイヤー盤上不在)と正確に同じ方向**(playtest-checklist 候補4に対応)。
3. **ジュース** — ダメージフロート、コイン飛散、松明の光、FLOOR CLEAR祝祭。触り心地の良さの予感。
4. 実写手持ちスマホ=モバイルUGC広告フォーマット(マーケ制作物の話であり、ゲーム本体の要件ではない)。

## 2. ギャップ分析(PV vs 現状)

現状のpresentation層(調査 2026-07-15):

- 盤面 = GridContainer + Panel/StyleBoxFlat の色分け(hidden=青灰 / revealed=明灰 / flag=暗赤 / detonated=黒)
- 数字 = デフォルトエンジンフォントの Label。フラグ = `_draw()` の三角形
- 敵 = セル右上の赤バッジ22×22px+「E」ラベル。プレイヤーはHUDテキストのみで盤上不在
- juice = 爆発セルのアルファフラッシュ + HUDラベルの色フラッシュの2種のみ。パーティクル/シェイク/フロートなし
- アセット(画像・フォント・音)完全ゼロ。presentation層 約1,130行(battle_screen 595 / cell_view 275 / main 134 / board_view 126)

**好材料**: battle_controller が型付きイベント(`mine_exploded` / `enemy_attacked` / `player_damaged` / `victory` 等)を発行する構造のため、演出強化は **domain / application に一切触れずテスト230件を壊さずに実現できる**。

### PV要素の3分類

| 分類 | 要素 | 着手可否 |
|------|------|----------|
| A. 演出・見た目 | ピクセルアートタイル、敵スプライト、ピクセルフォント数字、ダメージフロート、シェイク、爆発パーティクル、勝敗演出、SE | **今すぐ可**(presentation層のみ) |
| B. ゲーム構造 | 盤上プレイヤーアバターと移動、宝箱・装備・コイン、フロア連結ラン、敵の盤上移動 | **Phase 2以降**(handoff §3 先回り実装禁止。着手条件 = Phase 1 プレイテスト肯定) |
| C. マーケ制作物 | PV映像、ストア素材 | Aが済めば録画で作れる |

## 3. 推奨ゴール(PV完全模倣ではなく)

PVにはこのゲーム固有の一番の見せ場が映っていない — **「数字を読んで特定した地雷を敵に向けて起爆する」瞬間**。騎士の探索はよくある絵だが、起爆攻撃(地雷=攻撃資源)は本作だけの差別化点であり、核仮説そのもの。

→ **中間ゴール: 「実プレイ10秒の録画だけで起爆の気持ちよさが伝わる」状態**。
核仮説の検証(最優先タスク)と同じ方向を向き、PV制作(C)への布石(A)にもなる。

## 4. ロードマップ

### Step 0: 核仮説判定(現行最優先のまま・変更なし)

手動プレイテスト10項目(docs/playtest-checklist.md)の完了。演出投資はこの判定を覆さない範囲で並行可(課題1対応はプレイテスト品質を上げるため先行してよい)。

### Step 1: ジュース・パス(presentation層のみ/課題1対応候補1〜3を包含)

| タスク | 内容 |
|--------|------|
| 1-1 敵攻撃テレグラフ | カウントダウンを敵セル近傍に可視化。攻撃時: 敵セル発光 → 攻撃エフェクト(線 or 弾) → プレイヤーHP減少を連続演出で因果を1本に繋ぐ |
| 1-2 ダメージフロート | 被弾/与ダメの数値がその発生源から浮いて消える(「-2」等) |
| 1-3 起爆演出強化 | 爆心→隣接の時間差フラッシュ強化、パーティクル、軽い画面シェイク、ヒットストップ数フレーム |
| 1-4 HPバー化 | Player/Enemy HPをテキスト→バー+減少アニメに |
| 1-5 勝敗演出 | VICTORY/DEFEATオーバーレイに最低限の演出(フェード・スケール) |

**受入基準**: テスト230件 exit 0 / domain・application・config差分ゼロ / fixture・game_balance不変 / WSLgスクショ+実入力(`click:`)検証 / 課題1の「なぜHPが減ったか分からない」が解消したとユーザーが確認。

### Step 2: ビジュアルテーマ・パス(アセットパイプライン導入)

> **前提整備済み(2026-07-16 フェーズA〜D)**: 盤面は Node2D ワールド+Camera2D(パン/ズーム/12×12対応)へ移行済み。
> ビジュアル方針は **3/4 見下ろしビュー**で確定(decisions D34。アイソメ不採用)。タイル前面の厚み・背の高いキャラの
> Yソート重なりはプレースホルダ描画で実証済み。**アセットの座標契約・受け入れ条件は docs/view-spec.md §2/§6 が正本**
> (footprint 88px・足元=entity_anchor・texture_filter=NEAREST・ズーム離散化検討)。

| タスク | 内容 |
|--------|------|
| 2-1 アセット選定 | CC0/安価素材(Kenney、itch.io等)で**3/4見下ろしの**ダンジョンタイル(厚み付きhidden+床revealed)・敵スプライト・ピクセルフォントを選定(ユーザー確認を挟む) |
| 2-2 盤面テクスチャ化 | cell_node の StyleBoxFlat 描画 → テクスチャ(hidden=盛り上がりタイル / revealed=床)。数字をピクセルフォント化。CELL_INSET_PX=0 化 |
| 2-3 敵/プレイヤースプライト化 | プレースホルダトークン → スライム等のスプライト(足元アンカー・セルより背が高くてよい。可能ならアイドル2フレーム) |
| 2-4 SE導入 | 開放/フラグ/起爆/被弾/勝利の5点。BGMは任意 |
| 2-5 ライセンス記録 | 使用アセットの出所とライセンスを docs に記録 |

**受入基準**: Step 1と同じ+アセットのライセンス確認済み+view-spec §6 の条件(NEAREST・アンカー・数字可読性)。

### Step 3: ミニPV(仮説の安価な検証)

Step 1-2完了後の実プレイを30秒録画し、冒頭3秒に「起爆モーメント」を頭出しした構成で人に見せて反応を測る。実写手持ち風の本格PVはこの反応を見てから判断。

### Step 4: Phase 2(プレイテスト判定が肯定の場合のみ)

盤上プレイヤーアバター(課題1候補4 = PVの核心構図)、敵複数種・敵移動、ラン構造。**Phase 2設計時に本PVを北極星として仕様に組み込む**。handoff §16 バックログと統合して計画する。
