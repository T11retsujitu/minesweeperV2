# 設計判断の記録(Phase 1)

handoff の曖昧箇所・未決事項への裁定と、その理由・変更可能性。番号は docs/implementation-plan.md §2 に対応。

## ルール裁定

### D1. dud起爆(フラグ済み非地雷セルの起爆)= 安全開放として解決
**判断**: 1ターン消費、ダメージ0、セルは通常開放(0ならフラッド)、ログに dud 記録。
**理由**: 「起爆できるか否か」で地雷の有無が判別できてしまう情報リークを防ぐ。誤フラグのコストが「1ターン+情報」になり、不完全情報での起爆判断という核となる緊張感を保つ。
**変更可能性**: 高。dud を「何も起きない」にする案、ペナルティを付ける案もあり得る。プレイテストで判断。

### D2. 数字は生成時固定(起爆後も再計算しない)
**理由**: 標準マインスイーパーの慣習に一致し実装が単純。再計算すると起爆のたびに盤面情報がリセットされ推理の積み上げが崩れる。
**変更可能性**: 中。消費済み地雷を数字から除く方が推理しやすいという意見が出たら Phase 2 で再検討。

### D3. 爆発はセルを開放しない(ダメージのみ)
**理由**: 「開放=情報獲得」「起爆=攻撃」のリソース対立を純粋に保つ。爆発で周囲が開くと起爆が情報獲得も兼ねてしまい、開放行動の価値が下がる。

### D4. 起爆後セルのライフサイクル
**判断**: 起爆時にフラグ自動除去。detonated セルは以後タップ・フラグ・開放すべて無効。誤爆セルは revealed + detonated(明るい背景に X)。意図起爆セルは hidden + detonated(暗いクレーター)。
**理由**: 消費済み資源であることの視覚的明示と、状態遷移の単純化。

### D5. DETONATE はフラグ済みセルのみ受理 / プレビューは contains_mine 非依存
**理由**: UIフロー(フラグ→タップ→確認→起爆)とドメインの整合。プレビューが地雷有無に依存すると dud でも情報が漏れる。

### D6. 勝敗同時成立は勝利優先
**判断**: handoff §5.8 の順序を字義どおり実装。手順5(勝利判定)で確定したら敵カウント減以降(6-9)を実行しない。誤爆で敵と自分が同時に力尽きるケースはテストで勝利を検証済み。

### D7. リトライは同一盤面(fixed=同一fixture / random=同一seed)
**理由**: 「同じ盤面でやり直して学習する」体験と再現性(handoff §12 検証可能性)を優先。新しい盤面は New Seed で明示的に。

## 生成の裁定

### D8. ランダム盤面は敵3×3ゾーンに地雷「ちょうど3個」を保証
**判断**: handoff §5.2 は「最低1個」だが、敵HP6・隣接ダメージ2・通常攻撃なしの Phase 1 では、ゾーン地雷が2個以下だと**撃破不能盤面**が生成される。ちょうど3個 = ceil(6/2) を保証(≥1 を包含するので仕様準拠)。
**トレードオフ**: 全ランダム盤面で敵セルの数字が常に「3」になり多様性が下がる。Phase 2 で敵HPバリエーションや「3個以上」への緩和とあわせて再検討。

### D9. 敵のランダム配置は内側5×5(x,y∈[1,5])
**理由**: 盤端だとゾーンが8マス未満になり D8 の充足が難しくなる。内側配置でゾーン8マスを常に確保。

### D10. first_reveal_safe はランダム盤面のみ適用(handoff §4 からの変更)
**判断**: 固定盤面は座標が固定のため初手安全を保証できない(するとしたら盤面が変わってしまう)。ランダム盤面では、最初の REVEAL が地雷のとき該当地雷を決定的に移設(`hash(seed, タップ座標)` でシード)して数字を再計算。ゾーン地雷はゾーン内空きへ移設し D8 を維持。移設先から開放済みセル・敵セル・既存地雷を除外。
**注**: これは handoff §4 仮確定仕様 `first_reveal_safe: true` の適用範囲の限定であり、最終報告に明示済み。保証対象は「最初の REVEAL アクション」のみ(2手目以降の地雷開放は通常どおり誤爆)。

### D11. 固定盤面の initial_revealed は明示リストのみ開放(フラッドしない)
**理由**: fixture の初期情報量を設計者が正確に制御するため。fixture は初期開放セルに0が無いよう設計してあるので、プレイヤーから見て「0なのに開いていない」違和感は生じない。

### D12. 生成失敗時のフォールバック
試行上限100回(config)を超えたら fixture を使用し `used_fixture_fallback=true` で識別可能にする(handoff §5.2)。要求 seed は保持。

## 実装方針

### D13. ターン解決はイベント配列を返す純粋ドメイン
**理由**: UI・ログ・演出をロジックから分離(handoff §6.4「演出はゲーム状態を変更しない」)。テストがイベント列を直接検証できる。汎用イベントバス(autoload)は handoff §2 の禁止事項なので、イベントは resolver の戻り値のみ。

### D14. 入力ロックのプロトコル
controller はターン消費アクション受理時に `is_busy=true`、presentation が演出完了時に `notify_effects_done()` を呼んで解除。「解決中の追加入力拒否」(handoff §9)が演出時間中も含めて成立し、テストで解除前の入力拒否を検証。

### D15. 隣接計算(8近傍)と爆発半径の分離
隣接地雷数・フラッドはルール定義の固定8近傍(`ADJACENCY_RADIUS=1`)。爆発範囲は調整値 `EXPLOSION_RADIUS_CHEBYSHEV`。当初は共用していたが、爆発半径を調整すると盤面の数字まで変わる結合だったため分離した(レビューで検出)。

### D16. class_name 不使用・preload 統一 / autoload 不使用
**理由**: `.godot/` キャッシュが無い fresh clone でもヘッドレステストが決定的に動く。グローバル名前空間を汚さない。

### D17. Godot 4.4.1 + gl_compatibility レンダラ + 720×1280 canvas_items stretch
**理由**: WSLg で安定動作し将来の Android 移行(モバイルGPU)にも通用する。4.3互換構文で記述(typed Dictionary 等を回避)しフォールバック余地を残す。

### D18. 乱数はローカル RandomNumberGenerator + 明示 seed のみ
グローバル `randi()` 禁止。seed は state に保持し HUD 表示(固定盤面は `fixture: phase1_core_demo` 表示)。New Seed のみ時刻由来(非決定でよい唯一の箇所)。

### D19. 中心4ダメージは「爆心セルに敵がいる場合」として実装
通常プレイでは敵セル=安全マスなので発生しないが、ドメインは任意状態を受け付けテストで爆心4/隣接2/範囲外0を検証。敵が地雷上に乗り得る Phase 2(敵移動)への備え。

### D20. デバッグ機能のゲート
地雷表示・スクリーンショット・操作列実行は全て `OS.is_debug_build()` 内。リリースビルドのゲームルールに混入しない(handoff §10)。

### D21. 計画時ミニソルバーは出荷物に含まれない
fixture の論理解率検証(全解列挙)は計画フェーズの Python スクリプトで実施したもので、ゲーム本体にソルバーは実装していない(handoff §3「完全な論理解法保証ソルバー」は作らない、に準拠)。

### D22. アバタールールは ruleset 切替(単一 resolver 内分岐)で導入(Phase 2)
`combat_state.ruleset`("phase1" / "phase2_avatar"、デフォルト phase1)。全生成APIとcontrollerは末尾省略可能引数でデフォルト phase1 → **既存テスト230件は無変更のまま旧ルールを検証し続ける**。別resolverを作らないのは §5.8 共通尾部(countdown→攻撃→勝敗)の重複と乖離を避けるため。UIは常に avatar で起動。変更可能性: avatar が本流と確定したら phase1 分岐の削除を検討。

### D23. アバター移動・開放は8近傍(Chebyshev 1)
移動先=隣接の開放済み・非detonated・非敵セル。開放=隣接の未開放セル(プレイヤーは動かず周囲にタイルが開く=PVの構図)。0フラッドの連鎖は無制限のまま(隣接制約は起点のみ)。8近傍はプロジェクトの隣接ルール(D15 `ADJACENCY_RADIUS`)と一貫。

### D24. 起爆スプラッシュは意図起爆のみ・誤爆はフラット3のまま
DETONATE の爆風範囲にプレイヤーがいれば `explosion_damage_at` でダメージ(`source="detonation_splash"`、実質最大2=プレイヤーは地雷セルに立てない)。誤爆に爆風を加算すると常時3+2=5になり `ACCIDENTAL_MINE_DAMAGE=3` の意味が変質するため加算しない。位置取りリスクの導入が目的。

### D25. 移動もカウントダウンを進める(countdown 3 のまま)
移動=1ターン消費でターン圧力を生む。fixture攻略ライン(勝利7T・被弾2回・最終HP6 / 敗北5T)が countdown 3 で成立することを机上検証+walkthroughテストで固定 → **新バランス定数なし・game_balance.gd 差分ゼロ**。変更可能性: プレイテストで移動コストが重すぎる場合は「移動はカウント非消費」へ(resolver 1分岐)。

### D26. dud起爆の遠隔開放は仕様(移動詰みの脱出弁)
非地雷セルへのフラグ→起爆(dud)は従来から当該セルを遠隔開放する。アバタールールで移動経路が完全封鎖されても、1ターンと引き換えに遠隔で盤面情報を得る手段が残る=ハードロック不存在。test_avatar_rules で固定。

### D27. avatarランダム生成の player_start は盤面確定後に選出
地雷・敵配置の検証が通った後に、敵ゾーン外・非地雷セルから rng で決定的に選出し force_reveal。phase1 経路の rng 消費順は不変(同シード同盤面テスト保護)。初手安全化(D10)の移設プールは開放済みセルを除外するため player_start 領域へ地雷が移設されることはない。

### D28. クリア二層化: 敵撃破=PHASE_RECOVERY遷移、勝利は Perfect Clear か Finish で確定(Phase 2 / game-design §7.3)
avatar ルールのみ、敵HP0で `enemy_died + combat_won` を発行し `PHASE_RECOVERY` へ(phase1 は即 victory のまま凍結)。回収フェーズは REVEAL/MOVE/DETONATE/フラグ可・敵ステップ(カウントダウン/攻撃)全スキップ・誤爆死は通常敗北。全安全セル開放で `perfect_clear + victory{perfect:true}`(クレーター=地雷セルは妨げない)、`FINISH` アクション(recovery中のみ合法・ターン非消費)で `victory{perfect:false}`。同一アクションで両者死亡は D6 勝利優先を踏襲し **recovery に入らず** 即 victory{perfect:false}。報酬は祝祭演出+リザルト表示のみ(スコア・数値報酬なし=ユーザー決定 2026-07-16)。狙い: 「敵の周り以外を開ける理由がない」問題の解消(§7.3)。

### D29. 敵死亡後は敵セルへ移動可能(死体セル通行可)
`avatar_can_move_to` の敵セル除外を「敵生存中のみ」に変更。死んだ敵のセルを永久通行不可にすると回収フェーズの到達性を損なうため。回収の全安全セル到達性は test_recovery_phase のグリーディウォーカーで機械的に証明。

### D30. 回収フェーズ中は爆発の敵ダメージ計算をスキップ
敵は死亡済みのため `enemy_damaged` イベント自体を発行しない(HP負値や無意味な0ダメージイベントを避ける)。スプラッシュ・誤爆のプレイヤーダメージは通常適用(回収中の起爆も位置取りリスクを保つ)。

### D31. 縄張り=Chebyshev半径1・圏外はカウントダウン凍結式ポーズ(Phase 2 / game-design §7.4。**2026-07-16 改訂: 半径2→1**)
`TERRITORY_RADIUS = 1`(敵中心3×3)。プレイヤーが縄張り外にいる間はカウントダウンが**進まない**(`countdown_paused` イベント。ユーザー決定: 「攻撃無効」案ではなくカウント停止案)。カウント値は圏外でリセットせず**凍結**(残1のまま再侵入を迫る緊張を残す)。判定は**アクション解決後のプレイヤー位置**。敵攻撃はカウント減経由でのみ発火するため圏外必中は自然に消滅。
**半径の変遷**: 導入時は半径2(walkthrough 無変更で入る工学的利点+時間圧力維持)だったが、ユーザー実プレイ判断で**半径1に改訂**: (1) 5×5=25マスは 7×7 固定盤面(49マス)の過半を占め「どこにいても攻撃される」体感、(2) 敵隣接セルの開放には敵から距離2以内に立つ必要があり、半径2では「敵の隣の安全マスを開ける」行為が必ず被カウント圏内=所感「敵の隣が開けにくい」が構造的に未解決だった。半径1なら距離2(圏外)から無リスクで開放可能。
**トレードオフ(自覚済み)**: 時間圧力は大幅に弱まり実質「踏み込んだときだけ進む時計」になる(fixture 勝利ラインの被弾が2回→1回・最終HP6→8。無被弾ラインも構成可能)。12×12 では縄張りが盤面の6%まで縮小。プレイテストで圧力不足が出たら半径2へ戻すか敵複数で面圧を作る(いずれもオラクル再導出が必要)。phase1 は縄張り概念なし(常時進行)。

### D32. bump攻撃=1ダメージ+敵生存時のみ反撃2、ステップ6の特例なし(Phase 2 / game-design §7.4)
隣接(8近傍)の生存敵セルをタップで交戦(確認なし・移動と同格の直接動詞。bumpable ハイライトで予告)。`BUMP_DAMAGE=1` / `BUMP_COUNTER_DAMAGE=2`。撃破時は反撃なし=フィニッシャーの爽快感。交換レート最悪(1:2)+bump単騎は数学的不成立(番犬テストで固定)により**起爆優位を数値で担保**。カウントダウンは通常どおり進む(特例なし)— カウント残1でbumpすると反撃2+敵攻撃2を両方食らうのは「カウントを読んでbumpのタイミングを選ぶ」意図された駆け引き(攻撃スキップ特例はcountdown 0滞留と§5.8一様性の破壊のため不採用)。

### D33. 地雷除去(defuse)=隣接フラグ済み限定・敵位置無関係の2ダメージ・盤面が読み直せる
`DEFUSE_DAMAGE=2`(<=隣接爆発2)。自分の8近傍のフラグ済みセルのみ。地雷なら除去→開放→`recalculate_adjacent_mine_counts()`(周囲の数字が減る=推理のやり直し価値)→敵へ2(位置と無関係=「回収した弾薬を叩き込む」抽象化。recovery中はダメージなし)。自傷なし・クレーター化なし。誤フラグはdud扱い(フラグ解除+遠隔開放+0ダメージ)。起爆との役割分担: 起爆=遠隔・大ダメ・自傷/クレーターのリスク vs 除去=隣接まで歩く手数(縄張り内なら毎手カウント消費)を払う安全な換金。UIは起爆プレビューに Defuse ボタン(隣接時のみ)を追加した3択。

### D34. 盤面描画は Node2D+Camera2D、大型マップはカメラで、ビジュアルは 3/4 見下ろし(アイソメ不採用)(2026-07-16 ユーザー決定)
マップ拡大の方式として「カメラ導入で大型マップ」をユーザーが選択(9×9 への固定拡大やスクロールなし案は不採用)。これに伴い盤面を Control(GridContainer)から Node2D ワールド+Camera2D へ移行(公開 API は board_view 時代と同名維持)。ランダム戦は 12×12・地雷26(密度≈18.1%、7×7 の 18.4% を踏襲)。fixture は 7×7 のまま完全凍結。ドット絵アセットの見た目は **3/4 見下ろしビュー**(正方グリッド+タイル前面の厚み+背の高いキャラが上のセルに重なる)を採用し、**真アイソメトリック(菱形グリッド)は不採用** — 理由: (1) マインスイーパーは数字を読むゲームであり菱形化は盤面の可読性を下げる、(2) 入力判定・レイアウト・Zソート・アセット枚数のコストが大きい、(3) 奥行き感はスプライトの描き方(Yソート+厚み)でほぼ達成できる。座標アンカー等の描画契約は docs/view-spec.md が正本。

## ステータスラベルの意味論(UI 補完)

HUD 右下のステータスは「最後に実行したアクション/直近の重要イベント」の要約(Detonated / Accidental mine / Dud / Revealed / Flag on/off / Enemy attacked / VICTORY / DEFEAT)。詳細な履歴は下部のアクションログが正本。
