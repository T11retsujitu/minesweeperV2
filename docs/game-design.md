## 1. コンセプトと核仮説

**マインスイーパー × ダンジョンRPGの戦闘ローグライト。**
数字を読む推理で地雷の位置を特定し、その地雷を「敵への攻撃資源」として遠隔起爆して敵を倒す。

- **核仮説(検証中)**: 数字を読んで特定した地雷を、敵への攻撃として起爆する行為そのものが気持ちよいか
- **差別化点**: 地雷=脅威ではなく攻撃資源。「開けて避ける」ゲームではなく「特定して使う」ゲーム
- **北極星**: PVビジョン(docs/pv-vision-roadmap.md §1)—「盤上に主人公がいて、周囲にタイルが開き、数字と敵が見える」1カットで伝わる構図

## 2. 現行ルール(ruleset = "phase2_avatar"、UIデフォルト)

### 2.1 盤面と初期状態

| 項目 | 値 | 備考 |
|------|-----|------|
| 盤面 | 7×7、地雷9個 | `BOARD_W/H`, `MINE_COUNT` |
| 敵 | 1体、安全セル上に固定 | HP6・攻撃2・カウントダウン3 |
| プレイヤー | HP10、**盤上に位置を持つ** | 初期位置: fixture=(1,3) / ランダム=敵ゾーン外の安全セル(開放して出現) |
| 固定盤面 | `phase1_core_demo`(初期開放8マス) | 座標変更禁止(ソルバー検証済みオラクル) |
| ランダム盤面 | シード決定的。敵は内側5×5、敵周囲3×3に地雷ちょうど3個 | 初手安全移設あり(D10) |

### 2.2 プレイヤーの行動

| 行動 | ターン消費 | 条件・効果 |
|------|-----------|-----------|
| **開放(REVEAL)** | 1 | **自分の8近傍**の未開放セルのみ。0セルならフラッド連鎖(連鎖範囲は無制限)。地雷なら誤爆 |
| **移動(MOVE)** | 1 | **8近傍**の開放済み・非クレーター・非敵セルへ。開放済みセルのタップ=移動 |
| **起爆(DETONATE)** | 1 | **盤面のどこでも**フラグ済みセルを起爆可(遠隔=核仮説のメカニクス)。プレビュー→確認の2段階 |
| **bump攻撃(BUMP)** | 1 | 隣接の**生存敵セル**をタップで交戦。敵に1、**敵生存時のみ**反撃2(撃破時反撃なし=フィニッシャー)。カウントダウンは通常進行(D32) |
| **地雷除去(DEFUSE)** | 1 | **隣接のフラグ済みセル**のみ。地雷なら除去+開放+周囲の数字再計算+敵に2(自傷なし・クレーターなし)。誤フラグは dud 扱い。起爆プレビューの Defuse ボタンから(D33) |
| フラグ設置/解除 | 0 | 任意セル。長押し or 右クリック |

- **縄張り(D31)**: 敵中心 Chebyshev 半径2(`TERRITORY_RADIUS`、盤内クリップ)を盤面に色塗り表示。**プレイヤーが縄張り外にいる間はカウントダウンが進まない**(凍結。リセットなし)。判定はアクション解決後の位置。敵撃破の投資価値=時間圧力の空間化
- 合法手はUIで常時ハイライト(移動先=青緑 / 開放可能=黄)
- 非合法手(遠すぎる開放・不正な移動先)はターン非消費で拒否され、理由がステータスに出る

### 2.3 ダメージルール

| 事象 | ダメージ | 対象 |
|------|---------|------|
| 起爆の爆発 | 爆心4 / 隣接2(Chebyshev半径1の3×3) | 範囲内の敵 **とプレイヤー**(スプラッシュ、下記) |
| **起爆スプラッシュ** | 実質最大2(プレイヤーは地雷セル=爆心に立てない) | 意図起爆のみ。**位置取りのリスク**: 起爆前に爆風圏から離れるかの判断が生まれる |
| 誤爆(地雷を開放) | プレイヤーに**フラット3**(爆風加算なし、D24)。爆風は範囲内の敵に通常適用 | 誤爆セルはクレーター化(歩行不可) |
| dud(安全セルを起爆) | 0。当該セルが**遠隔開放**される | 肩透かしだが、移動封鎖時の脱出弁として機能(D26) |
| 敵の攻撃 | 2 | カウントダウン0で発動、カウントは3に戻る |

### 2.4 ターン解決順序(§5.8準拠・不変)

1. 入力確定 → 2. 開放/起爆/移動を解決 → 3. 爆発ダメージ(敵→プレイヤーspalshの順)→ 4. 敵死亡 → 5. **勝利判定(成立なら以降スキップ=勝利優先)** → 6. カウントダウン減 → 7. 0なら敵攻撃 → 8. プレイヤー死亡 → 9. 敗北判定 → 10. UI同期

- **移動もカウントダウンを進める**(D25)= ターン圧力。1手の価値が「情報を取る/位置を直す/攻撃する」の三択になる
- 勝利: 敵HP0 / 敗北: プレイヤーHP0(同時成立は勝利優先、D6)

### 2.4b クリア二層化(§7.3 実装済み、D28〜D30)

- 敵HP0(avatar)= 即勝利ではなく **回収フェーズ(PHASE_RECOVERY)** へ(`combat_won` イベント)。カウントダウン・敵攻撃は停止
- 回収フェーズ: 開放・移動・起爆・フラグを継続可能(ターンは進む)。誤爆・スプラッシュのダメージは通常適用され、HP0なら**通常敗北**(押し引きの実弾)
- **全安全セル開放 → Perfect Clear**(`perfect_clear` + `victory{perfect:true}`)。クレーター(地雷セル)は妨げない
- **Finish(任意終了)** → `victory{perfect:false}`。ターン非消費
- 同一アクションで両者死亡は D6 勝利優先(recovery に入らず即勝利)
- 敵死亡後は敵セルへ移動可能(死体セル通行可、D29)
- Perfect Clear の報酬は祝祭演出+リザルト表示のみ(スコア・数値報酬なし)

### 2.5 詰みが存在しないことの保証

勝利条件は敵撃破であり全盤面開放は不要。移動経路がクレーター等で完全封鎖されても、フラグ+起爆(dud)による遠隔開放で情報取得が続行できる(D26)。敗北はHP0のみ。

### 2.6 固定盤面の攻略オラクル(tests/test_avatar_walkthrough.gd で全手固定)

- **勝利ライン(7ターン、最終HP6)**: 起爆(1,1)→起爆(0,1)→開放(1,4)[被弾-2]→開放(2,4)→**移動(2,4)**→開放(1,5)[11マスフラッド、被弾-2]→起爆(0,3)で敵撃破(クリア二層化後は T7 で回収フェーズへ遷移。回収の全安全セル到達性は test_recovery_phase のウォーカーで証明)
- **敗北ライン(5ターン)**: 誤爆(0,3)→移動(2,2)→誤爆(1,1)[被弾-2]→移動(3,2)→誤爆(4,1)で敗北
- 設計意図: 勝利には推理(部分集合推理を含む)と移動による経路選択の両方が必要。誤爆3回+敵攻撃1回でちょうど敗北ラインに乗る

## 4. 体験レイヤ(ジュース・パス、pv-vision-roadmap Step 1 完了)

ルールではないが体験の一部として記録。すべてイベント購読のみ(domain非依存)で実装:

- **因果の可視化**: 敵バッジにカウントダウン数字(残1で「1!」点滅)→ 攻撃時は敵セル発光→弾がアバターへ飛翔→着弾でフロート+HPバー減少。「なぜHPが減ったか」が1本の演出で繋がる(課題1対応)

- HPバー(減少痕ゴースト付き)、勝敗オーバーレイのフェード/スケール演出
- アセットは全て描画プリミティブの仮置き(画像・音は生成AIで後日差し替え予定)

### ルール変更の意図(Phase 1 → アバター)

1. **空間の意味付け**: 盤面が「情報の表」から「自分が立つ場所」になり、敵との距離・爆風との距離が実感を持つ
2. **1手の重み**: 移動がターンを消費するため「どこから情報を取るか」の経路計画が推理に加わる
3. **リスクの対価**: 遠隔起爆は無傷の攻撃だったが、スプラッシュ導入で「安全圏に下がってから起爆する」位置取りの駆け引きが生まれる
4. **核仮説は不変**: 「数字を読んで地雷を特定し敵に向けて起爆する」ループ自体には手を加えていない(遠隔起爆維持=R5)

## 7. Phase 2 設計方針(2026-07-16 ユーザーと議論のうえ確定)

> 背景: PV例(pv-vision-roadmap §1)と現状ビルドの比較分析から「なぜ現状は無味乾燥に見えるか」を議論し、
> 敵・アイテム・スキル・クリア条件の存在理由を一つの基準に統一した。以降の Phase 2 実装はこの方針に従う。

### 7.1 設計の背骨: 新要素 = 「素のマインスイーパの構造的弱点へのパッチ」

マインスイーパの核の動詞は「**リスクを払って情報を買う**」。素のマインスイーパには4つの構造的弱点があり、
本作の追加要素はすべて**この弱点を埋める装置**として定義する(戦闘ゲームの接ぎ木ではない):

| # | 弱点 | パッチとなる要素 |
|---|------|----------------|
| ① | 時間圧力がない(いくらでも長考できる) | **敵**のカウントダウン = 長考のコスト化。1手の価値が生まれる |
| ② | 失敗が二値(1ミス即死) | **HP制** = 誤爆が死ではなくダメージになる(実装済み) |
| ③ | 運ゲーの瞬間(推理で割り切れない50/50) | **アイテム/スキル** = 資源を払って局面を割り切る脱出弁(スキャン・地雷除去等) |
| ④ | 報酬が「死なないこと」しかない | **敵=地雷の売り先**(特定した地雷が攻撃資源になる)+ クリアボーナス(§7.3) |

**新要素の採否基準**: 「情報・ターン・HP・地雷(弾薬)のあいだに新しい交換レートを作るか?」
作らない要素は飾りなので追加しない。

### 7.2 UI/UXテーマ: 「盤面が読める風景」

無味乾燥さの最大因は「盤面が絵として死んでいる」こと(大半が同色の閉じタイルで、プレイヤーの知識・危険・
資源が画面に描かれない)。以下を UI/UX の統一テーマとする:

- **プレイヤーの知識を盤面に描く**: フラグ済みセル=爆弾スプライト表示(推理の成果が武器として並ぶ)
- **危険を盤面に描く**: 敵の縄張り(§7.4)を色で塗る。数字の色分けによる危険度ヒートマップ
- **ポジティブフィードバック**: 安全開放・フラッド連鎖・敵撃破・クリアボーナスで祝祭的な演出を返す
  (現状は被ダメージ=ネガティブしか弾けていない)
- 盤面の開放率が上がる方向のルール/バランス調整を優先(閉じた盤は情報ゼロの絵)

### 7.3 クリア条件: 二層構造(基本=敵全滅 / 全開放=ボーナス)

- **基本条件(勝利)**: 敵全滅(現行どおり。核仮説の主役=起爆が勝利に直結する構図を維持)
- **上乗せ(ボーナス)**: 敵撃破後も盤面を開け続け、全安全セル開放で「Perfect Clear」ボーナス
- **敵を倒すメリットの定義**: 敵撃破=**カウントダウン(時間圧力)の停止**。以降は落ち着いて残り盤面を
  開けてボーナスを回収できる。つまり戦闘は義務ではなく「安全と利益を買う投資」。将来のラン構造では
  敵はドロップ(動く宝箱)も持つ
- 効果: 「敵の周り以外を開ける理由がない → 盤面の大半が死ぬ」問題を解消し、マインスイーパ本来の
  プッシュ・ユア・ラック(もう1マス開けるか?)を欲張りループとして接続する
- 「出口到達」型のクリアはラン構造(Phase 4)導入時に再検討。今は扱わない

### 7.4 敵の扱い: 縄張りモデル(移動AIは保留)

- **縄張り(攻撃圏)**: 敵は固定のまま、周囲N×N を縄張りとして**盤面に色で描く**。プレイヤーが縄張りの
  外にいる限り攻撃されない(またはカウントダウンが進まない)。中に入るとカウントダウン進行
  - 解決する問題: 「敵が飛び道具に見える」(位置と無関係の必中攻撃)/「移動に防御的価値がない」
  - 敵の近くの数字を読みに行く=縄張りに踏み込む、というリスクの対価が自然に生まれる
- **敵の盤上移動は保留**: 1ターン1マスの追跡は遅い時計か釣り餌にしかなりにくい。縄張りモデルで空間の
  意味を先に確立し、必要になれば「縄張りごと動く敵」として上位互換で導入する
- **近接戦(bump攻撃)**: プレイヤーが敵マスへ移動=交戦、を導入候補とする。常時使える攻撃動詞になる。
  ただし**起爆が常に「賢い選択」であるバランスを厳守**(例: bump=低ダメージ+反撃あり / 起爆=高ダメージ
  +反撃なし)。bumpはトドメ・緊急手段の位置づけ
- **地雷除去スキル**: 起爆との役割分担で導入候補とする
  - 起爆(遠隔): 大ダメージ / スプラッシュ自傷リスク / セルがクレーター化
  - 除去(隣接8マスのみ): 小〜中ダメージ / 自傷なし / クレーター化しない / 地雷の隣まで歩くリスクを払う
  - 「特定した地雷をどう使うか」の遠隔/近接二択となり、地雷=資源のアイデンティティを深める

### 7.5 実装優先順(採用決定済み: 下記 1〜3。敵移動は不採用・保留)

1. **「盤面が読める風景」化**(§7.2): フラグ→爆弾表示・縄張りの色塗りは presentation 層中心で先行可能
2. **クリア二層化**(§7.3): 敵撃破→カウントダウン停止→回収フェーズ+Perfect Clear ボーナス。
   単発戦闘プロトタイプのまま game_balance 追加程度で安価に検証できる
3. **攻撃動詞の追加**(§7.4): bump攻撃・地雷除去(隣接版)。起爆の優位性は数値で担保
4. ポジティブフィードバック演出はすべての段に並行して足す(presentation 層のみ)

- 実装時の不変条件は llm-status.md §7 に従う(fixture 座標不変・数値は game_balance.gd 一元化・
  レイヤ依存方向)。ルール変更を伴う段(2・3)は decisions.md に D28 以降として判断を記録すること
