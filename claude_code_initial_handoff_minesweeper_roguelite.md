> **用途**: Claude Code に新規プロジェクトの実装を開始させるための初期指示書  
> **対象**: Claude Code / エージェントコーディング  
> **文書ステータス**: Phase 1 実装開始用  
> **最優先事項**: ゲーム全体を作ることではなく、コアループが面白いかを検証できる戦闘プロトタイプを完成させること

---

## 0. あなたの役割

あなたは本プロジェクトのリードゲームエンジニアとして行動する。

企画をそのまま機械的に実装するのではなく、以下を同時に満たす実装を行うこと。

1. 最短距離でプレイ可能なプロトタイプを作る
2. ゲームルールと表示ロジックを分離し、後から調整しやすくする
3. 自動テストとデバッグ手段を用意し、仕様変更による破損を検出できるようにする
4. 不明点を理由に停止せず、可逆的な仮決定を置いて記録する
5. コア仮説の検証に不要な機能を作らない

**計画書だけを作って終了してはならない。**  
リポジトリを確認し、実装・テスト・実行確認・ドキュメント更新まで行うこと。

ただし、Phase 1 の完了前にローグライト全体、ショップ、恒久成長、広告、課金へ進んではならない。

## 1. プロジェクト概要

### 一文企画

> マインスイーパーの推理で地雷を特定し、敵を爆破して進む短時間ローグライト。

通常のマインスイーパーでは、地雷は避ける対象である。  
本作では、地雷を敵にダメージを与える有限の攻撃資源として扱う。

プレイヤーは数字から地雷位置を推測し、地雷をフラグで特定して任意起爆する。安全マスを開けて情報を増やすほど推理は確実になるが、行動するたびに敵の攻撃カウントが進む。

### プレイヤーに発生させたい判断

- さらに盤面を開けて確実な情報を得るか
- 不完全な情報のまま起爆へ移るか
- どの地雷を、どの敵に、どの順序で使うか

### 最初に検証する仮説

> 数字を読んで特定した地雷を、敵への攻撃として起爆する行為そのものが気持ちよいか。

この仮説が成立しない場合、遺物、ビルド、メタ進行、演出を追加しても企画は成立しない。

## 2. 開発方針

### 想定技術スタック

```yaml
engine: Godot 4.x
language: GDScript
primary_platform: Android
orientation: portrait
runtime: offline
server: none
repository_policy:
  preserve_existing_stack: true
  do_not_migrate_without_approval: true
```

リポジトリに既存のエンジン、言語、規約がある場合は、まずそれを調査すること。既存構成が合理的なら上記より優先する。空のリポジトリまたは技術未決定の場合は Godot 4.x + GDScript を採用する。

### 実装原則

- モバイル実機を前提とするが、開発中はPC上でも操作可能にする
- UIは仮素材でよい。外部アセット探索を優先しない
- ゲームルールをUIノードに直接埋め込まない
- 乱数は必ずシードを指定・表示・再現可能にする
- 数値は可能な限り設定またはデータとして切り出す
- 過剰な抽象化、将来用フレームワーク、汎用イベントバスを先に作らない
- 1つの状態を複数ノードが独立して保持しない
- 警告やエラーを放置しない

## 4. Phase 1 の仮確定仕様

企画上の未決事項について、最初の試作では以下を採用する。

```yaml
prototype:
  board_width: 7
  board_height: 7
  mine_count: 9
  enemy_count: 1
  enemy_moves: false
  player_exists_on_board: false
  enemy_action_system: countdown
  flag_action_consumes_turn: false
  reveal_action_consumes_turn: true
  detonate_action_consumes_turn: true
  first_reveal_safe: true
  chain_explosion_enabled: false
  deliberate_detonation_self_damage: false
  accidental_detonation_self_damage: true
  weak_basic_attack: false
  solvable_board_required: false
```

### この仮決定の意図

- 最初の試作では、地雷起爆だけで敵を倒せるようにし、通常攻撃へ逃げない
- 敵移動はゲーム性を増やすが、最初の検証では変数を増やしすぎるため無効化する
- フラグ操作を無料にし、スマートフォンでの入力負荷を下げる
- 完全な論理解法保証は重要だが、最初からソルバーを作ると検証開始が遅れる
- ランダム盤面だけでは評価しにくいため、必ず手作りの固定盤面を用意する

## 5. 戦闘ルール

### 5.1 盤面

各セルは最低限、以下の状態を持つ。

```yaml
cell:
  coordinate: Vector2i
  contains_mine: bool
  adjacent_mine_count: int
  reveal_state: hidden | revealed
  flag_state: none | flagged
  detonation_state: intact | detonated
```

敵の状態はセルの状態と分離する。敵は座標を持つが、セルそのものではない。

### 5.2 初期盤面

- 盤面サイズは7×7
- 地雷数は9
- 敵は1体
- 敵の初期位置は安全マスとする
- 敵位置のセルは戦闘開始時から開放状態とする
- 敵の周囲3×3内に最低1個の地雷が存在するように生成する
- 最初にプレイヤーが開くセルは必ず安全にする
- 固定盤面では、少なくとも1個の地雷を論理的に特定できる初期情報を与える

ランダム生成に失敗した場合は無限ループさせず、試行上限後にフォールバック盤面を使用する。

### 5.3 プレイヤー操作

#### 通常タップ

- 未開放かつ未フラグのセル: 開放する
- 開放済みセル: 何もしない
- フラグ済みセル: 即時起爆せず、起爆確認状態へ移行する

#### 長押し

- 未開放セルのフラグを設置または解除する
- フラグ操作ではターンを進めない
- 開放済みセルにはフラグを置けない

#### 起爆確認

フラグ済みセルを通常タップした場合、以下を表示する。

- 爆発範囲
- 爆心地と周辺の予想ダメージ
- 影響を受ける敵
- 起爆ボタン
- キャンセルボタン

起爆ボタンを押した時点で1ターン消費する。

PC上では、左クリックをタップ、右クリックを長押し相当として扱ってもよい。

### 5.4 マス開放

安全セルを開いた場合:

- 周囲8セルの地雷数を表示する
- 数字が0なら、隣接する安全セルを再帰またはキュー方式で自動開放する
- 自動開放全体を1回のプレイヤー行動として扱う

地雷セルを直接開いた場合:

- その地雷を即時爆発させる
- プレイヤーへ誤爆ダメージを与える
- 爆発範囲内の敵にも通常どおりダメージを与える
- 1ターン消費する

### 5.5 爆発

初期数値は以下とする。調整値として一箇所に集約すること。

```yaml
explosion:
  center_damage: 4
  adjacent_damage: 2
  radius_chebyshev: 1
  chain_explosion: false
  consumed_after_detonation: true
```

爆発範囲は起爆セルを中心とした3×3である。

- 爆心地座標の敵: 4ダメージ
- 周囲8マスの敵: 2ダメージ
- 範囲外: 0ダメージ
- 地雷は起爆後に消費済みとなり、再起爆できない
- 範囲内の別地雷は誘爆しない
- 意図的な起爆ではプレイヤーはダメージを受けない

### 5.6 敵

Phase 1 の敵は1種類のみとする。

```yaml
enemy:
  id: basic_soldier
  max_hp: 6
  attack_damage: 2
  initial_countdown: 3
  reset_countdown_after_attack: 3
  moves: false
```

- 敵は盤面上の1座標に固定される
- 敵の残り行動カウントを常時表示する
- ターンを消費するプレイヤー行動の後にカウントを1減らす
- カウントが0になったらプレイヤーへ攻撃する
- 攻撃後、カウントを3へ戻す
- 敵HPが0以下なら、そのターンに敵攻撃を行わない

### 5.7 プレイヤー

```yaml
player:
  max_hp: 10
  accidental_mine_damage: 3
```

- HPが0以下で敗北
- 敵を倒した時点で勝利
- Phase 1 では回復、シールド、通常攻撃、スキルを実装しない

### 5.8 ターン解決順序

ターンを消費する行動は、必ず次の順序で解決する。

1. プレイヤー入力を確定
2. マス開放または起爆を解決
3. 爆発ダメージを解決
4. 敵死亡を解決
5. 勝利判定
6. 勝利していなければ敵カウントを減らす
7. 敵カウントが0なら敵攻撃
8. プレイヤー死亡を解決
9. 敗北判定
10. UIを最終状態へ同期

同一入力から複数回ターンが進むバグを避けるため、ターン解決中は入力をロックする。

## 6. UI要件

### 6.1 画面構成

```text
┌────────────────────────┐
│ Player HP   Enemy HP    │
│ Enemy intent: Attack 2  │
│ Seed: 12345             │
├────────────────────────┤
│                        │
│        7×7 Board        │
│                        │
├────────────────────────┤
│ Mode / Help / Retry     │
│ Action Log              │
└────────────────────────┘
```

### 6.2 視認性

- 数字は色だけに依存せず、文字そのものを明瞭に表示する
- 敵がいるセルでも数字が読めること
- 敵はセル中央を完全に覆わないこと
- フラグ、地雷、爆発済み、選択中、爆発プレビューを明確に区別する
- 盤面は一般的なスマートフォン縦画面で無理なく収まること
- タップ対象は小さくしすぎないこと
- 起爆は通常開放と視覚的に明確に分離すること

### 6.3 仮素材

Phase 1 では以下でよい。

- `Control`、`GridContainer`、`Button`、`Label`、`ColorRect`等の標準UI
- 単純な図形または記号による敵・地雷・フラグ表示
- Tweenまたは短い標準アニメーションによる爆発フィードバック

外部の著作物やライセンス不明素材を追加しない。

### 6.4 最低限の演出

- 地雷起爆時に爆心地と周辺セルを短時間強調
- 敵被弾時にHP変化が分かる
- プレイヤー被弾時にHP変化が分かる
- 勝利・敗北をモーダルまたはオーバーレイで表示

演出はゲーム状態を変更してはならない。ロジック完了後に表示する。

## 8. 固定テスト盤面

ランダム性に左右されずコアループを評価できるよう、固定盤面を1つ用意する。

固定盤面はコード内の読みにくい真偽値配列ではなく、座標リストまたは人間が読める形式で定義する。

例:

```yaml
fixture_id: phase1_core_demo
board_size: [7, 7]
mines:
  - [1, 1]
  - [3, 1]
  - [5, 1]
  - [2, 3]
  - [4, 3]
  - [1, 5]
  - [3, 5]
  - [5, 5]
  - [6, 3]
enemy:
  position: [3, 3]
  hp: 6
initial_revealed:
  - [3, 3]
  - [3, 2]
  - [2, 2]
```

上記座標は例であり、そのまま採用する義務はない。次の条件を満たすよう調整すること。

- 最初から少なくとも1個の地雷を論理的に推定できる
- その地雷を起爆すると敵へダメージが入る
- 1個の起爆だけでは敵が倒れず、もう一度推理が必要になる
- 敵の初回攻撃前に、急げば1回以上起爆できる
- コアループを30秒から2分程度で確認できる

## 9. 自動テスト要件

外部テストプラグインを導入しなくても実行できる構成を優先する。既存プロジェクトにテスト基盤がある場合はそれを使う。

最低限、ヘッドレス実行可能なテスト入口を用意する。

想定コマンド例:

```bash
godot --headless --path . -s res://tests/run_tests.gd
```

環境上のGodot実行ファイル名が異なる場合は、READMEへ実際のコマンドを記載する。

### 必須テスト

#### 盤面

- 四隅、辺、中央の隣接地雷数が正しい
- 0セル開放で連結する安全セルが展開される
- フラグ済みセルは通常開放されない
- 同じシードから同じ地雷配置が生成される
- 地雷数が設定値と一致する
- 敵位置が安全セルである
- 敵周囲に攻撃可能な地雷が最低1個存在する

#### 操作・ターン

- フラグ設置ではターンが進まない
- 安全セル開放でターンが1だけ進む
- 起爆でターンが1だけ進む
- 1入力で敵カウントが複数減らない
- ターン解決中の追加入力が拒否される

#### 爆発

- 爆心地へ4ダメージ
- 隣接マスへ2ダメージ
- 範囲外へ0ダメージ
- 別の地雷を誘爆しない
- 起爆済み地雷を再起爆できない
- 意図的な起爆でプレイヤーがダメージを受けない
- 地雷直接開放でプレイヤーへ3ダメージ

#### 勝敗

- 敵HPが0以下なら勝利
- 敵を倒したターンには敵攻撃が発生しない
- プレイヤーHPが0以下なら敗北
- リトライで状態が初期化される

テストがUI描画に依存しないようにすること。

## 10. デバッグ要件

### 常時確認できる情報

開発ビルドでは以下を画面またはデバッグパネルで確認可能にする。

- 現在のシード
- プレイヤーHP
- 敵HP
- 敵カウント
- 消費ターン数
- 現在の入力モード
- 最後に実行したアクション

### アクションログ

最低限、以下を記録する。

```text
Turn 1: reveal (2, 4) -> safe, adjacent=1
Turn 2: detonate (3, 2) -> enemy damage=2
Enemy countdown: 2 -> 1
Turn 3: reveal (1, 1) -> accidental mine
Player damage: 3, HP=7
```

### デバッグ操作

開発ビルドに限り、以下があると望ましい。

- 地雷表示のON/OFF
- 固定盤面とランダム盤面の切り替え
- 同一シードで再生成
- 新しいシードで再生成

デバッグ機能がリリース用ゲームルールへ混入しないようにする。

## 11. 実装手順

以下の順序で作業すること。

### Step 1: リポジトリ調査

- ファイル構成
- 使用エンジンと言語
- 既存README、CLAUDE.md、規約
- Git状態
- 実行方法
- テスト基盤
- 既存アセットとライセンス

無関係なファイルを変更しない。

### Step 2: 実装計画を記録

`docs/implementation-plan.md` を作成し、以下を記載する。

- 現状
- 採用アーキテクチャ
- 作成・変更予定ファイル
- 実装順
- テスト方針
- 未確定事項と仮決定

計画を作った後、そのまま実装へ進む。

### Step 3: ドメインロジック

最初にUIなしで以下を完成させる。

- 盤面モデル
- 隣接地雷数
- セル開放
- ゼロセル展開
- フラグ
- 起爆
- 爆発範囲
- 敵とプレイヤーのダメージ
- ターン解決
- 勝敗

### Step 4: 固定盤面とテスト

- 固定盤面を追加
- 必須テストを追加
- ヘッドレスで実行
- 失敗を修正

### Step 5: UI接続

- 盤面表示
- タップ、長押し、PC代替入力
- 起爆プレビューと確認
- HP、敵カウント、シード表示
- 勝敗とリトライ

### Step 6: ランダム生成

- シード付き生成
- 初手安全
- 敵位置安全
- 敵周囲の攻撃可能性保証
- 試行上限とフォールバック

### Step 7: 実行確認

- 固定盤面で勝利までプレイ可能
- 誤爆から敗北まで確認可能
- リトライ可能
- 同一シード再現可能
- UI操作で二重ターンが発生しない

10. 最適行動が毎回「盤面をほぼ全部開く」になっていないか

## 14. 守るべき制約

### Git・ファイル操作

- 既存のユーザー変更を上書きしない
- 無関係なリファクタリングをしない
- 大量削除や履歴改変をしない
- `git reset --hard`、強制push、無断pushをしない
- 秘密情報を追加しない
- ビルド生成物を無秩序にコミットしない

### 依存関係

- Phase 1のためだけに大型プラグインを導入しない
- 新規依存を追加する場合は、必要性、ライセンス、代替案を記録する
- アセットは自作の仮素材または明確に利用可能なものだけを使う

### 仕様判断

- 仕様が曖昧でも、実装可能な仮決定を置いて前へ進む
- 仮決定は `docs/decisions.md` に理由と変更可能性を書く
- コア仮説に影響する重大な変更は、実装前または報告時に明示する
- 面白さを保証したと断定しない。実装とプレイテストを分けて扱う

## 15. 作業完了時の報告形式

作業終了時は、以下の形式で報告すること。

```markdown
## 実装結果

### 完了した内容
- ...

### 主な設計判断
- ...

### 変更した主要ファイル
- `path/to/file`: 役割

### 実行確認
- 起動確認: 成功 / 未確認
- 固定盤面プレイ: 成功 / 未確認
- ランダム盤面プレイ: 成功 / 未確認

### テスト
- 実行コマンド: `...`
- 結果: X件成功 / Y件失敗

### 既知の制約
- ...

### 次に検証すべき点
- ...
```

実行できなかった項目を成功扱いにしないこと。環境不足で確認できない場合は、実装済みと実行確認済みを明確に区別する。
